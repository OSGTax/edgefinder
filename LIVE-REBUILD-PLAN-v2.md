## Architecture

END-TO-END (matches LIVE-REBUILD-PLAN.md + the existing agent/broker.py scaffold; codebase facts verified against the repo 2026-06-25; all vendor costs re-grounded 2026-06-25 — see costTable sources).

ACCOUNT OF RECORD = Alpaca paper account (base https://paper-api.alpaca.markets, default $100k, separate keys). Cash, positions, fills, P&L are read live from Alpaca via GET /v2/account, /v2/positions, /v2/orders (wrapped in agent/broker.py). The owner can log into Alpaca's own dashboard and see the identical orders/positions the agent created — the 'verifiable' guarantee is verifiable OWNERSHIP of PAPER orders, NOT verified real-time NBBO execution; paper fills are an Alpaca simulation.

THREE RUNTIMES — with order SUBMISSION resolved to Render (the prior plan's biggest architectural contradiction). NOTE: this resolution is a DESIGN, not yet built — verdict/risks say DESIGNED-NOT-BUILT, and Phase 1 must land the primitive + a crash-mid-submit reconcile test before any GO.
1) Render always-on FastAPI streamer (Starter $7/mo on a free Hobby workspace — Hobby confirmed free with NO mandatory seat fee for a single Starter instance, kuberns/costbench 2026; Free spins down after 15-min idle). It is the ONLY persistent actor and owns: ONE Alpaca market-data WebSocket; an in-memory live-quote cache; SSE fan-out to /desk over a NEW /api/desk/stream; /api/desk/portfolio served from Alpaca live account+positions; AND — DESIGNED — ORDER SUBMISSION. The Routine does NOT hold the broker socket or submit directly; it writes an intended-orders record (client_order_id stamped) and signals an authenticated Render endpoint, and Render submits to Alpaca and reconciles. UNVERIFIED PREREQUISITE (now a Phase-2 GO gate, not assumed): the Routine sandbox runs under a DEFAULT DOMAIN ALLOWLIST (grounded: cloud Routines run in a sandboxed network with a default allowlist; SKILL.md today reaches only Supabase over HTTPS). The custom Render *.com endpoint MUST be added to the Routine allowlist AND proven reachable from a real Routine run BEFORE the submit-via-Render handshake is declared sound — if Routines restrict outbound egress to the allowlist and the endpoint can't be added, the handshake silently fails and NO order places. This closes the STATELESS-SANDBOX hazard ONLY once proven: a Routine that dies after deciding leaves NO half-placed order, because the always-on actor reconciles every intended-order against Alpaca by client_order_id before (re)submitting (idempotent). The streamer launches as an asyncio task in the FastAPI lifespan (render_start.py calls uvicorn.run(app, workers=1); the task lives in app.py's lifespan). MANDATORY resilience: WS + every browser EventSource break on every Render deploy/instance replacement and on WS drop (SIGTERM ~60-90s after the new instance is healthy), so both need reconnect+backoff, boot-time cache warm, quote-staleness detection (fall back to Alpaca REST latest-quote, never act/display off a stale tick). DEPLOY-WINDOW HAZARD (newly explicit): on a single-instance Starter (workers=1, no second instance) every deploy is a total stream + submission-endpoint outage for the SIGTERM window; an intended-order signaled DURING a deploy must be retried by the Routine/Render with the SAME client_order_id (idempotent), and the guard loop is dead during the restart — the staleness banner must surface 'streamer restarting'. Blocking alpaca-py REST runs in a threadpool, off the async SSE/WS loop.
2) Claude Code Routine (HOURLY, market hours) = the discretionary brain, fresh stateless clone (no local persistence, no missed-run catch-up). Reads Alpaca positions + live quotes (off the Render cache or REST) + Polygon backing data, decides, WRITES intended orders (client_order_id) + signals Render to submit, and records each suggestion stamped with live price+timestamp+evidence into desk_thinking/desk_decisions/desk_journal. EVERY run FIRST pulls Alpaca state and rebuilds the desk_* mirror (self-healing of the DISPLAY; heals the mirror, NOT a missed DECISION or an intra-hour move — those are Decision 1). agent/broker.build_order()/submit() take NO client_order_id and there is NO cancel() (verified) — both net-new, landed+tested in Phase 1 as a HARD PRECONDITION of Phase 1 GO. Market-calendar-aware no-op outside session via broker.is_market_open() (a get_clock() boolean — authoritative for the SUBMIT gate incl. half-days, but NOT half-day-schedule-aware for cadence planning). A no-op run still burns one of Max's 15 daily runs.
3) Server-side guard loop on Render (deterministic) = the ONLY thing that can react between hourly runs. Decision 1: IN scope (a HARD PRECONDITION of Phase 5 GO) or the charter carries an OWNER-SIGNED 'book is unmanaged between hourly runs' acceptance BEFORE the live flag flips. If IN, the guard loop stamps its own client_order_id namespace (prefix 'guard-') so guard-flattens JOIN cleanly as system-originated. The guard loop is a FIFTH concurrent responsibility on the 0.5-vCPU process (WS + cache + SSE + submission + guard) — explicitly part of the load test.

OFF-STREAMER OPS (moved OFF the single 0.5-vCPU process so the monitor does not die with the thing it monitors): the liveness watchdog AND the Supabase keep-alive write run on GitHub Actions cron (2,000 free Linux min/mo), NOT as asyncio tasks inside the streamer lifespan. OWNER-PAGING CHANNEL (newly named): the watchdog projects a skipped-hour / dead-WS / dead-streamer condition onto a deduped GitHub issue (label 'edgefinder-alert', the repo's existing retired alerts.py pattern reused) and auto-closes on recovery — same mechanism the old liveness.yml used with the workflow GITHUB_TOKEN. A streamer crash still leaves an independent external actor to page the owner and keep Supabase warm.

DATA FLOW: Alpaca SIP WebSocket -> Render quote cache -> SSE -> browser (owner-marks default) and -> Routine reads. SYMBOL-UNIVERSE BUDGET (newly stated): the streamer subscribes to held + watchlist + active-candidate names, capped at a STREAMED-SYMBOL BUDGET of ~50 (held ~10-20 + a ~30-name watchlist) — chosen because (a) the free Basic feed caps at 30 symbols/1 connection so the $87 IEX shell cannot watch a top-100 list, and (b) SSE/egress and the load test both scale with symbol count. Top-N screening uses Polygon daily-bar/REST snapshots, NOT the live WS, so the live socket never needs the full universe. Polygon/Massive -> nightly backing-data ingest -> Supabase hot set + R2 Parquet archive -> backtest tool. desk_* tables are a DISPLAY MIRROR reconciled FROM Alpaca each cycle; per-fill rationale/run_id JOINED to Alpaca fills by client_order_id (namespaced 'routine-' / 'guard-'). An Alpaca fill with NO matching desk_ rationale = system-originated (guard flatten); a rationale with no fill = reject/partial. ledger.rebuild_positions int(shares) must move to FRACTIONAL before any notional order. A NET-NEW desk_iv_history table persists the daily Alpaca/Massive options-snapshot IV per held/watched name so IV-rank/percentile can be self-computed (the future tool's storage target). SCHEMA (newly specified): one row per (underlying_symbol, snapshot_date) storing ATM implied_volatility + the 30-day-ATM IV the agent reads for rank, plus delta/gamma/theta/vega of the ATM contract; per-UNDERLYING (not full chain) to keep writes ~50 rows/day; retain 504 trading days (~2yr) for a 1yr+ IV-rank window with headroom — well within Supabase Free.

MASTER SWITCH: settings.live_trading_enabled (config/settings.py:110), overridable as EDGEFINDER_LIVE_TRADING_ENABLED. There is NO 'EDGEFINDER_LIVE' var.

REASONING SPLIT: Alpaca = present (live quotes, positions, fills, OPRA options IV/Greeks). Polygon/R2 + backtest tool = past. FRED + Alpha Vantage + the daily options-IV snapshot + desk_iv_history = future.
